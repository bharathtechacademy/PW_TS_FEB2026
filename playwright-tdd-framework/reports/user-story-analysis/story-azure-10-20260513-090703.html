DescriptionAs an existing user, I want to log in to my Creatio CRM account using my business email and password through a secure and well-designed form and get the errors when we update invalid credentials.
Acceptance Criteria
  • Business Email field should be highlighted red and show Invalid email format when the user enters an invalid email format or invalid domain names.
  • Business Email field should be highlighted red and show Invalid email format when the user enters an email longer than 400 characters.
  • Password field should be highlighted red and show Invalid value. Value must contain up to 100 characters when the user enters a password longer than 100 characters.
  • Invalid email or password should be displayed at the bottom of the page when the user enters an invalid username/password combination and attempts to log in.

Requirement Interpretation

Plain-language summary: this story requires a login form for existing Creatio CRM users with client-side and/or server-side validation that blocks malformed email values, overly long inputs, and invalid credentials, while presenting clear inline and page-level errors.

In Scope

  • Login form display for business email and password.
  • Email-format validation, including invalid domain patterns and overlength inputs.
  • Password length validation up to 100 characters.
  • Authentication error messaging for incorrect credentials.
  • Red error highlighting for invalid fields and error copy consistency.

Out of Scope

  • Password reset, account registration, and MFA flows.
  • Role-based access after authentication.
  • Browser/device-specific rendering beyond normal responsive behavior.
  • Security hardening not described by the story, such as lockout policy or CAPTCHA.

Assumptions and Open Questions

Assumption: the login form includes at least two fields, Business Email and Password, plus a submit action.
Assumption: the validation copy must match the acceptance criteria exactly, including capitalization and punctuation, unless product design specifies otherwise.
Open question: should invalid email format be blocked on blur, on input, or only on submit?
Open question: is the 400-character email limit inclusive of the full address string, and should the UI truncate or simply reject it?
Open question: does password length validation happen only client-side, or must the backend return the same message for consistency?
Open question: should spaces be trimmed from the email field before validation and authentication?

Risks and Dependencies

Risk 1: validation copy or trigger timing may differ between frontend and backend, causing unstable tests if the implementation is not standardized.
Risk 2: the authentication service may return a generic error message instead of the exact text required by the story, which would break strict verification.
Risk 3: email and password length constraints could differ between UI, API, and database layers, producing boundary defects.
Dependency: test execution needs stable credentials for at least one valid user and a controllable invalid-user dataset.
Dependency: the login service must be reachable in the chosen test environment; otherwise only UI-level validation can be verified.

Functional Test Plan

Objectives

  • Verify that valid users can log in successfully.
  • Verify that invalid email and password values are rejected with the expected messages.
  • Verify boundary behavior at the 400-character email limit and 100-character password limit.
  • Verify that the page-level invalid-credentials message appears only when authentication fails.

Test Levels

  • UI: field validation, error rendering, and message placement.
  • Integration: auth handshake and backend response mapping.
  • API: optional login-service verification if a direct endpoint is available in the environment.

Test Data Strategy

  • One known-good business email and password pair.
  • One invalid but well-formed email, one malformed email, and one overlength email.
  • One password at exactly 100 characters and one longer than 100 characters.
  • One invalid credential pair to trigger the bottom-of-page authentication error.

Environment Prerequisites

  • Creatio CRM login environment is available and stable.
  • At least one valid test account exists and is not locked.
  • Validation messages and selectors are known or discoverable in the target build.

Entry Criteria

  • Story and acceptance criteria are approved or ready for test execution.
  • Test environment is deployed and accessible.
  • Test accounts and input data are prepared.

Exit Criteria

  • All P0 and P1 scenarios pass or have approved defects.
  • Validation behavior matches the acceptance criteria.
  • No open blocker prevents login validation from being trusted in the target build.
