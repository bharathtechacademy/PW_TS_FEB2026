Requirement Interpretation

User Story

"As an existing user, when I attempt to log in, I want to receive error messages for invalid credentials so that I can correct my input and successfully access my account."

Business Context

This story addresses the need for proper input validation and user feedback on the login form for Creatio CRM. Users should receive clear, actionable error messages when they provide invalid credentials or improperly formatted data. This is part of the broader "User Login Experience" epic and helps ensure a smooth, user-friendly authentication process.

Acceptance Criteria

1 Email Field - Invalid Format/Domain

The Business Email field should be highlighted with red color and display 'Invalid email format' when an invalid email format or domain is entered.

2 Email Field - Length Validation

The Business Email field should be highlighted with red color and display 'Invalid email format' when the email exceeds 400 characters.

3 Password Field - Length Validation

The Password field should be highlighted with red color and display 'Invalid value. Value must contain up to 100 characters' when the password exceeds 100 characters.

4 Invalid Credentials - Error Message

An 'Invalid email or password' error should be displayed at the bottom of the page when invalid username/password is entered.

Scope Analysis

In-Scope

  • Email format validation (RFC 5322 compliance)
  • Email length validation (max 400 characters)
  • Password length validation (max 100 characters)
  • Invalid credentials error message display
  • Visual field highlighting (red color)
  • Error message text and positioning

Out-of-Scope

  • Password strength validation
  • Two-factor authentication
  • Account lockout after failed attempts
  • Remember me functionality
  • Social login integration
  • Password reset functionality

Assumptions and Open Questions

Assumptions

A1: Email format validation will follow RFC 5322 standard for valid email addresses.
A2: Field highlighting with red color refers to both border and background or just border. Current assumption: border color.
A3: The error messages should appear immediately on field blur or after attempting login.
A4: "Invalid email or password" message should not specify which field is incorrect for security reasons.
A5: The login form will be the default entry point for the application.
A6: All validation rules apply to both desktop and mobile views.

Open Questions

  • Q1: Should email validation be case-insensitive?
  • Q2: Should the form show validation errors on submit only or in real-time as user types?
  • Q3: How should the system handle SQL injection attempts in these fields?
  • Q4: What happens if password field contains special characters or unicode?
  • Q5: Should there be a maximum number of failed login attempts before lockout?
  • Q6: Are there any specific domains that should be whitelisted/blacklisted?

Risks and Dependencies

Risks

R1: Email Format Complexity (Medium Risk)

RFC 5322 standard for email validation is complex. Oversimplified regex patterns may reject valid emails or accept invalid ones. Recommend using a dedicated email validation library.

R2: Security Vulnerability - Information Disclosure (Medium Risk)

Generic error message "Invalid email or password" helps prevent user enumeration attacks, but developers might be tempted to show specific errors. Need code review to ensure compliance.

R3: Cross-Browser Compatibility (Low Risk)

Field highlighting and error display may render differently across browsers. Need testing on Chrome, Firefox, Safari, Edge, and mobile browsers.

R4: Accessibility Compliance (Medium Risk)

Color-only highlighting (red) is not accessible for colorblind users. Should include ARIA labels and additional visual indicators beyond color.

R5: Performance - Validation on Every Keystroke (Low Risk)

Real-time validation could impact performance if not properly debounced. Recommend client-side debouncing.

Dependencies

  • D1: Backend API must be available and responsive for credential validation
  • D2: Database of valid user accounts must be accessible and properly indexed
  • D3: UI component library or design system for consistent styling and error display
  • D4: Frontend framework/technology (React, Vue, Angular, etc.) must support real-time validation
  • D5: Test environment with test user accounts and various invalid credentials
  • D6: Browser automation tools (Playwright) for UI testing
  • D7: Compliance with OWASP security standards for authentication
